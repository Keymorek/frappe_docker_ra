本文档用于指导 `fashion_erp` 的人工业务验收。

目标有 3 个：

1. 给出一条可直接执行的人工验收主线
2. 明确每个功能点的“本轮通过标准”和“最终成功标准”
3. 把所有已规划功能都纳入验收视野，包括：
   - `已完成基础版`
   - `部分完成`
   - `外部依赖阻塞/暂停`
   - `下一步研发计划`

## 3. 验收前准备

### 3.1 环境准备

1. 执行站点 `migrate`
2. 清缓存并刷新 Workspace
3. 使用 `System Manager` 或具备完整权限的业务测试账号
4. 确认 `Fashion ERP` Workspace 可见

### 3.2 基础数据准备

建议至少准备以下测试数据：

| 数据类型 | 最低准备要求 |
|---|---|
| 品牌 | 1 个启用品牌，且已维护简称 |
| 商品类目 | 至少 2 个 `Style Category Template`，建议 1 个上装、1 个裤装 |

| 颜色 / 颜色组 | 至少 2 个颜色组、3 个颜色 |
| 尺码体系 / 尺码编码 | 至少 `TOP`、`BOTTOM` 两套，并各自有多条尺码 |
| 成品物料组 | 至少 1 个成品物料组 |
| 仓区 / 库位 / 库存状态 | 各至少 1 条启用数据 |
| 供应商 | 至少 1 个原辅料供应商、1 个外包工厂 |

| 客户 / 公司 / 渠道店铺 | 各至少 1 条 |
| 销售订单测试货品 | 至少 1 个通过 `Style` 生成的成品 SKU |

## 4. 推荐人工验收主线

建议按下面 5 条主线执行。这样既能覆盖功能，又能减少重复录入。

### 4.1 主线 A：商品开发 -> 打样 -> 工艺 -> 外包下单

步骤：

1. 在 `Style` 新建一个款号
2. 录入品牌、商品类目、季节、年份、主面料、里料、成品物料组
3. 录入 2 个颜色
4. 选择 `size_system`，再在 `Style Size` 中只选择本款需要的尺码
5. 保存后检查一级到四级类目、类目全路径是否自动回显
6. 点击 `创建模板货品`
7. 点击 `生成单品编码`
8. 点击 `查看矩阵`
9. 反向测试：若已经生成 SKU，尝试修改 `size_system` 或 `Style Size`
10. 从 `Style` 直接创建 `Sample Ticket`
11. 让打样单走完 `新建 -> 已下发 -> 打样中 -> 待评审 -> 已确认`
12. 从 `Style` 或相关入口创建 `Craft Sheet`
13. 反向测试：若打样单未确认，尝试发布工艺单
14. 发布工艺单
15. 从工艺单继续创建 `Outsource Order`

本轮通过标准：

- `Style` 可正常保存，必填和 Link 校验生效
- 类目能按 `1-4` 级路径搜索，保存后可回显路径
- `size_system` 会受类目规则约束
- SKU 只按已选 `Style Size` 生成，不按全体系展开
- 已生成 SKU 后，直接修改 `size_system / Style Size` 会被拦截
- 打样单状态动作、日志留痕可用
- 工艺单发布前会校验打样单是否已确认
- 工艺单可以作为外包下单依据

最终成功标准：

- 商品开发链路从 `Style` 到 `Outsource Order` 可稳定闭环
- 不再依赖表外 Excel 维护类目、尺码、主里料基础信息

### 4.2 主线 B：原辅料采购 -> 供料视图 -> 外包到货 -> 质检落账

步骤：

1. 建立面料、辅料、包装耗材物料
2. 建立原辅料供应商和外包工厂
3. 在外包单上录入 `materials` 子表
4. 打开外包单的供料视图，观察 `required / on_hand / on_order / prepared / to_purchase`
5. 新建 `Purchase Order`，让其引用外包单备货上下文
6. 提交采购单
7. 新建 `Purchase Receipt`，检查款号、外包单、供料上下文是否自动带出
8. 回到供料视图，确认 `on_order / on_hand` 等数量有变化
9. 新建 `Outsource Receipt`
10. 录入正常到货数量，并确认收货
11. 生成 `QC_PENDING` 待检入库草稿
12. 绑定待检入库凭证并确认已入库
13. 在到货明细上填写 `SELLABLE / REPAIR / DEFECTIVE / FROZEN` 分配
14. 生成最终质检落账草稿并确认质检完成
15. 反向测试：增加一条“零到货 + 短装数量”的异常行

本轮通过标准：

- 原辅料和包装耗材可走采购、收货、入库主线
- 外包供料视图能显示净需求和待采购口径
- `Purchase Receipt` 能回填外包上下文
- 外包到货单可从收货推进到待检入库、再到质检完成
- 质检分配数量有校验，不允许非法数量
- 异常行可以留痕，且不会被错误带入入库或质检落账

最终成功标准：

- 外包到货前后的供给、入库、质检、异常数据可以完整追踪
- 可作为库存、到货、成本报表的数据基础

### 4.3 主线 C：订单履约 -> 发货 -> 包装耗材 -> 履约成本

步骤：

1. 先用手工方式准备一张 `Sales Order`
2. 检查订单头和订单行是否具备履约状态字段
3. 从销售订单执行 `配货 -> 拣货 -> 打包`
4. 观察订单头和订单行状态是否推进
5. 从销售订单生成 `Delivery Note` 草稿
6. 在出货单上挂包装耗材
7. 生成包装耗材 `Stock Entry(Material Issue)` 草稿
8. 在 `Delivery Note` 上录入手工快递费
9. 提交 `Delivery Note`
10. 检查销售订单头、订单行、履约状态、已发数量是否回写
11. 取消 `Delivery Note`
12. 再次检查销售订单头和订单行状态是否回退

本轮通过标准：

- 销售订单可按 `配货 / 拣货 / 打包` 推进履约状态
- `Delivery Note` 可以由订单生成，并在提交/取消时回刷订单状态
- 包装耗材可挂单并生成出库草稿
- 手工快递费能纳入履约总成本
- 当前系统不会把“业务锁库存”误显示为真实 WMS 级锁库

最终成功标准：

- 订单履约状态、发货状态、包装耗材成本、快递费成本都能沉淀到单据和报表

### 4.4 主线 D：售后 -> 退款 / 补发 -> 库存闭环 -> 订单闭环

步骤：

1. 基于已有 `Sales Order / Delivery Note` 建立 `After Sales Ticket`
2. 检查原订单、发货单、明细行是否能正确关联
3. 走 `待退回 -> 已收货 -> 质检中 -> 待处理`
4. 退款场景：
   - 申请退款
   - 完成退款
   - 检查关单逻辑
5. 补发场景：
   - 从售后单直接创建补发 `Sales Order`
   - 让补发单走正常履约流程
   - 检查售后单是否自动回写补发履约状态
   - 检查补发单完成后售后单是否自动关闭
6. 库存闭环场景：
   - 从售后单直接提交待检入库凭证
   - 再直接提交最终处理凭证
   - 检查售后单 `库存闭环状态 / 待检入库凭证 / 最终处理凭证`
7. 反向测试：
   - 未完成最终库存处理时尝试关闭工单
   - 补发未真实履约完成时尝试关闭工单
   - 撤销最终处理凭证，检查工单是否回退

本轮通过标准：

- 售后工单不是纯备注单，能驱动退款、补发、库存和订单状态回写
- 补发单可从售后单直接创建并保存
- 补发单履约完成后会自动推动售后关闭
- 售后库存凭证可直接提交，不再只停在草稿
- 结案条件真实生效，不能提前手工关单

最终成功标准：

- 售后链可把订单、补发、库存三条线闭合
- 售后结果可进入订单状态、库存状态和报表口径

### 4.5 主线 E：报表与看板

步骤：

1. 打开 `6. 运营报表`
2. 逐一打开以下报表并执行查询：
   - `Style Inventory Overview`
   - `Material Supply Overview`
   - `Outsource Receipt Overview`
   - `Sales Fulfillment Overview`
   - `After Sales Overview`
3. 打开 `7. 财务与成本`
4. 逐一打开以下报表并执行查询：
   - `Outsource Estimated Cost Analysis`
   - `Material Procurement Cost Analysis`
   - `Fulfillment Cost Analysis`
5. 打开 `9. 辅助生产`
6. 打开 `Production Board`
7. 抽查 2 至 3 个单据结果，核对报表口径与源单是否一致

本轮通过标准：

- 9 张报表 / 看板都能打开、筛选、返回结果
- 关键字段与源单据一致，不出现明显错口径
- Workspace 入口位置清晰，业务用户能按流程找到报表

最终成功标准：

- 业务用户可直接从工作台查看库存、供料、到货、履约、售后、成本和生产看板结果

### 6.3 本轮通过标准

- 所有外部平台依赖项都被统一标记为 `外部依赖阻塞/暂停`
- 不存在“文档说阻塞、界面却像正式可用”的情况
- 验收报告中对这些功能的结论统一使用 `阻塞`，而不是 `不通过`

## 8. 本轮通过门槛建议

建议把本轮是否可进入更大范围业务试用，定义为以下条件同时满足：

1. `主线 A` 到 `主线 E` 中所有 `已完成基础版` 功能全部 `通过`
2. 所有 `部分完成` 功能的缺口都被明确记录
3. 所有 `外部依赖阻塞/暂停` 功能都被统一标记为 `阻塞`

5. 不存在因为文档、入口或字段文案造成的业务误解

- `当前已交付范围通过`

- `当前已交付范围不通过`
