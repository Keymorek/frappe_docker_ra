## 一、业务边界冻结

当前业务链已经明确为：

`季度主题与设计理念（外部软件） -> 款式录入 -> 打样与工艺单 -> 外包下单与预计成本 -> 商品入库 -> 仓储履约 -> 售后闭环 -> 运营/财务报表`

同时，供应链和成本口径也已经明确：

1. `季度主题 / 设计理念 / 品类规划` 仍在外部软件，不进入本 ERP
2. 本 ERP 的正式起点是 `款式录入`
3. 电商平台 API 由于资质问题无法接入，因此 `平台接口同步永久搁置`
4. 订单、售后、发货状态当前只能 `手工同步`
5. 大货生产由 `第三方外包工厂` 完成
6. 内部生产流水线只承担 `打样辅助`
7. 服装生产所需 `面料 / 辅料` 由我方提供，必须纳入采购、备货和库存管理
8. 发货时使用的 `包装耗材` 也由我方采购、入库、出库
9. 外包下单与原料出库 `不做自动联动`，只要求订单与原料建立人工可追踪关联
10. 快递费先采用 `手工录入金额`
11. 因此，系统不是完整制造 ERP，而是 `商品开发 + 原辅料供给 + 外包供应链 + 仓储履约 + 售后财务` 的 ERP

## 二、产品定位

`fashion_erp` 当前应定义为：

`面向女装电商公司的商品开发、打样协同、原辅料备货、外包下单、入库履约、售后闭环与经营分析系统`

系统的核心价值不在车间排产，而在以下 8 件事：

1. 款式与 SKU 主数据稳定
2. 打样和工艺单有据可查
3. 原辅料和包装耗材的采购、备货和库存可追踪
4. 外包下单与预计成本可追踪
5. 商品入库与库存状态可控
6. 仓储履约和售后闭环清晰
7. 快递费、包装耗材和货品成本能一起进入经营核算
8. 运营与财务报表能反映真实经营情况

## 三、ERP 的真正起点和终点

### ERP 之前

以下内容不放进当前 ERP：

1. 季度主题讨论
2. 设计理念确认
3. 季度品类分布确认

这些仍由外部软件承接。

### ERP 之内

ERP 需要承接以下完整链路：

1. `Style` 款式录入
2. 打样流程
3. 工艺单管理
4. 原辅料与包装耗材采购、入库、备货
5. 向外包工厂下单
6. 预计成本记录
7. 商品入库
8. 仓储履约
9. 售后闭环
10. 运营与财务报表

## 四、核心用户角色

建议当前围绕这几类角色设计：

1. 商品/设计
   - 录入款式、维护款式信息、确认工艺要求
2. 打样/版房
   - 跟踪样衣、确认工艺单、沉淀样品结论
3. 采购/跟单
   - 采购面辅料和耗材、向外包工厂下单、跟进交期、记录预计成本和异常
4. 仓库
   - 收货、质检、上架、拣货、打包、发货、退货处理
5. 售后/运营
   - 手工同步订单、处理售后、补发、退款
6. 财务/管理层
   - 看成本、库存、履约、售后和经营报表

## 五、核心业务流程分析

### 1. 款式录入

`Style 建档 -> 颜色/尺码体系 -> SKU 生成`

目标：

- 商品在进入打样和下单前已经结构化

### 2. 打样与工艺单

`发起打样 -> 样衣跟踪 -> 工艺单确认 -> 样品结论`

目标：

- 让打样不是口头沟通，而是可追踪对象
- 让后续外包下单能直接引用工艺单

### 3. 原辅料与包装耗材供给

`采购 -> 到货入库 -> 备货 -> 引用到外包订单或履约环节`

目标：

- 原辅料采购和库存可追踪
- 包装耗材采购、入库、出库可追踪
- 不自动驱动生产出库，但要知道原料被哪张外包订单引用
- 当前供料主线应由 `Outsource Order` 驱动，而不是 `Production Ticket`
- 第一阶段先建立轻量供料视图，而不是直接上完整 `Supply Plan`

### 4. 外包下单与预计成本

`选择工厂 -> 下单 -> 引用工艺单 -> 引用原辅料 -> 记录预计成本 -> 跟进交期`

目标：

- 让外包下单成为核心单据
- 让商品预计成本从一开始就进入系统
- 让原辅料与外包订单形成可追踪关系

### 5. 商品入库

`外包工厂送货 -> 收货 -> 质检 -> 入最终库存状态`

目标：

- 库存来源明确是外包到货
- 商品不直接跳过质检进入可售

### 6. 仓储履约

`订单手工同步 -> 配货 -> 拣货 -> 打包 -> 发货`

目标：

- 在没有平台 API 的情况下，先保证内部履约链路可执行
- 包装耗材能在履约时挂到出货单并计入成本
- 快递费可以手工录入并进入履约成本

### 7. 售后闭环

`售后申请 -> 收货 -> 质检 -> 退款/补发 -> 库存回写 -> 结案`

目标：

- 把客服动作、库存动作、订单动作串起来

### 8. 运营与财务报表

目标：

- 能看到库存、履约、售后、预计成本、采购/外包情况
- 能看到包装耗材和快递费等履约成本

## 六、当前最重要的问题

从产品角度看，当前最需要解决的不是“如何把生产做复杂”，而是：

1. 款式和 SKU 是否足够稳定
2. 打样和工艺单是否能沉淀
3. 原辅料和包装耗材是否可追踪
4. 外包下单与预计成本是否可控
5. 到货入库是否可追踪
6. 订单履约成本是否真实
7. 售后和库存是否闭环
8. 管理层是否能看到真实经营数据

## 九、建议的产品架构

当前阶段建议拆成 7 个工作域：

1. `商品中心`
   - 款式、颜色、尺码、SKU、价格、状态
2. `原辅料与耗材中心`
   - 面料、辅料、包装耗材、采购、入库、备货、占用引用
3. `打样与工艺中心`
   - 打样、工艺单、样品确认
4. `外包供应链中心`
   - 外包下单、预计成本、原料引用、到货、异常
5. `仓储履约中心`
   - 收货、质检、上架、配货、拣货、打包、发货、包装耗材、快递费
6. `售后中心`
   - 售后工单、退款、补发、库存回写
7. `经营分析中心`
   - 运营报表、财务报表、成本与库存分析

## 十、路线建议

### 阶段 A：基础已完成

对应现有第一期和第二阶段已完成部分：

1. 商品与 SKU
2. 库位和库存状态
3. 售后基础工单
4. 打样单基础版

### 阶段 B：当前应立即收口

主题：

`供料联动 + 质量保障`

目标：

1. 让采购、收货与 `外包备货` 场景形成服务端联动
2. 为当前状态流转补第一轮测试基础

### 阶段 C：随后进入

主题：

`手工订单同步 + 履约成本`

目标：

1. 手工同步订单进入系统并留痕
2. 履约状态可推进
3. 包装耗材和快递费能进入履约成本

### 阶段 C-2：质量保障与性能收口

主题：

`测试基础设施 + 状态机测试 + 数据访问优化`

目标：

1. 为核心工具函数和服务层建立正式测试基础
2. 为关键状态流转建立回归测试
3. 收口当前明显的 N+1 查询和重复加载问题

### 阶段 D：随后进入

主题：

`售后闭环 + 运营财务`

目标：

1. 售后与订单闭环规则落地
2. 运营与财务报表可用

### 阶段 E：后置

主题：

`内部生产深化`

目标：

1. 只在打样辅助确有需要时，再扩展内部生产对象
2. 不做完整制造 ERP

## 十一、产品计划建议

接下来建议按下面顺序推进：

1. 先做 `采购/收货与外包备货联动`
2. 再做 `测试基础设施与状态机测试`
3. 再做 `手工订单同步与履约状态`
4. 再做 `包装耗材挂单与快递费`
5. 再做 `售后与订单闭环规则`
6. 再做 `性能收口`
7. 最后做 `运营与财务报表`

## 十三、改写后纳入的规划说明

以下原建议已经改写后正式纳入计划：

1. 原 `Production Ticket` 驱动的物料需求计算
   - 已改写为 `Outsource Order` 驱动的供料需求与待采购视图
2. 原 `Supply Plan` 方案
   - 已改写为先在 `Outsource Order.materials` 上补轻量视图与聚合逻辑
3. 原 “Client Script 自动更新供料计划”
   - 已改写为“前端只做默认值，关键聚合与回写由服务端 / hook 驱动”

以下内容继续忽略，不纳入当前计划：

- 工厂虚拟仓
- 自动发料到外包工厂的库存联动
- 以 `Production Ticket` 为大货主轴的供料计划

## 十二、结论

当前项目的正确路线应是：

`款式录入 -> 打样 -> 工艺单 -> 原辅料备货 -> 外包下单 -> 商品入库 -> 仓储履约与履约成本 -> 售后闭环 -> 运营/财务报表`

这条链路才符合你当前公司的真实经营方式。
